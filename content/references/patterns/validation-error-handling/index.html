Form errors

When a user is filling out a form, they should be notified as soon as possible to errors so they can be corrected within the flow of their task rather than being collected until they think they are finished. If possible a field should be validated when focus has left the field. Sometimes this is not possible, in which case the validation will run when the form is submitted.

A validation error should be accompanied by a message letting the user know how to correct the error. Generally this message will be one of two types, although there will certainly be cases where a different message is needed:

  • <Item> is required.
  • Please enter a valid <item>.

It is sometimes the case that a form is quite long and the field that is triggering an error is scrolled out of view. To help the user identify what needs to be corrected, autoscroll to the first field that is causing an error.

In cases where a field is validated on focus loss, re-run the validation for a field triggering an error if the field value has been changed. This will ensure that the user immediately knows whether their change will be successful.

Page errors

There are a variety of page-level errors that can happen, usually due to a system failure rather than a user action. Generic error messaging is available for these scenarios, although the content can be customized if it needs to be more precise or prescriptive.

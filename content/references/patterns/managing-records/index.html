Add records

There are three general scenarios for adding records.

From Lists or Tiles

Adding a record is often an important action for a list or tile. In these cases, use an add button on the left side of the action bar for that component.

From Lookup Fields

When you need to allow the user to add records on the fly when they are performing a lookup, use an add button to the right of the lookup field.

From Collection Fields

When you need to allow the user to add values of a field type quickly and the data that needs to be entered is simple, use an add link below the list of already-added records.

Edit Records

In general, editing a record should happen in a modal form to allow the user to focus on the task of making the edit. The complexity of the form will depend on the nature of the data being edited, and the context in which an edit is likely to take place. If the user will tend to make a targeted change a concise form with just the necessary fields can be used. If there is a likelihood that the user will make a set of changes, or will need to reference related information while making a change, the full record-edit form should be used.

Remove Records

There are several scenarios for removing records.

Remove Records Stored in the Database

The operation of removing a record will differ depending on the nature of the information.

Delete

Use delete when it's okay to permanently delete a record. Keep in mind that features that reference the deleted item will not function as usual.

When deleting an item, confirm the action with a toast saying “<Item name> has been deleted.”

Archive

Use archive when an item will no longer be used, but where features will need to continue to reference the archived item. When an item is archived, the state is not permanent (the item can be become un-archived via "restore"). Features that reference the archived item will function as usual.

When archiving an item, confirm the action with a toast saying “<Item name> has been archived.”

Deactivate

Deactivate lives somewhere between archiving and deleting. You can deactivate an item where you no longer want features tied to it to continue to function going forward, but where you want past usage of the item to still function. The state is not permanent (the item can be become active again via "restore"). Features that reference the deactivated item will not function as usual.

When deactivating an item, confirm the action with a toast saying “<Item name> has been deactivated.”

Ignore

Ignore can be used when an item from the system no longer needs to be used, but the item cannot be deleted from the system for auditing or technical reasons. The state is not permanent (the item can be become un-ignored via "restore"). Features that reference the ignored item will not function as usual.

When ignoring an item, confirm the action with a toast saying “<Item name> has been ignored.”

Remove Ancillary Information

There are some cases where the items being removed are either not true records or are ancillary to the main record. In these cases the cost of accidentally removing the item is low, so there is no need for the user to confirm the action or to display a toast after the action is complete.
